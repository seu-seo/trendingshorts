## TL;DR (먼저 결론)

V0는 **"보이는 것"은 깔끔하지만 "보이지 않는 곳"이 약하다.** 코드 자체는 초보자 수준 치고 잘 정리됐으나, 발표 데모로서는 세 가지 큰 위험을 안고 있다:

1. **외부 의존성 3개가 동시에 살아있어야 화면이 돈다** — Tailwind CDN, Anthropic API, 발표 현장 Wi-Fi. 셋 중 하나라도 흔들리면 데모가 멈춘다.
2. **매니페스토 핵심 가치 #2 "감이 아닌 데이터로 결정"과 사실상 어긋난다** — 숫자(8.2%, +39%)는 화면에 표시되지만 그 숫자가 결과물의 형태/품질에 영향을 주지 않는다. "데이터를 보여준 vibes 데모"다.
3. **사용자가 처음 화면을 봤을 때 던질 질문 5개에 답을 못 한다** — 이건 발표장에서 교수/심사위원이 똑같이 던질 질문이다.

## 1. 코드 품질

### 잘된 점 (간단히)
- 모듈 구분(HTML/CSS/JS 3파일)과 섹션 주석은 학습용으로 모범적이다.
- `escapeHtml()` — innerHTML로 모델 출력을 그릴 때 XSS 방지를 미리 깔아둔 건 의식 있는 결정.
- `Promise.allSettled` 사용 — 한 톤 실패가 다른 톤을 죽이지 않는 부분 실패 처리는 V0치고 좋은 선택.
- `sk-ant-` prefix 검증으로 명백한 오타를 즉시 잡는 것.

### 개선해야 할 점 (발표 전에)

**(a) 응답 구조 가정이 너무 낙관적이다 — `script.js:137`**
```js
return data.content[0].text;
```
`content`가 빈 배열이거나 첫 요소가 `tool_use` 같은 다른 타입이면 그대로 터진다. `stop_reason`이 `max_tokens`인 경우(잘림)도 무시된다. 발표 현장에서 모델이 한 번 변덕 부리면 끝난다.
→ 최소한 `if (!data.content?.[0]?.text) throw new Error("응답 형식 예상과 다름")` 한 줄.

**(b) `parseScript()`가 부분적인 마크다운만 잡는다 — `script.js:144-151`**
```js
cleaned = cleaned.replace(/^```json\s*/i, "").replace(/^```\s*/, "");
cleaned = cleaned.replace(/```\s*$/, "");
```
모델이 `여기 JSON입니다:\n\`\`\`json\n{...}\n\`\`\`` 처럼 prefix 텍스트를 붙이면 시작 부분 정규식이 안 맞고 `JSON.parse` 실패. 시연 도중 한 번 일어나면 그 톤은 빨갛게 뜬다.
→ 대안: `text.match(/\{[\s\S]*\}/)` 로 첫 `{`부터 마지막 `}`까지를 슬라이스해서 파싱하는 게 더 견고하다.

**(c) JSON 스키마 검증이 사실상 없다 — `script.js:178`**
```js
const bodyArr = Array.isArray(script.body) ? script.body : [String(script.body)];
```
`body`가 배열이 아닐 때 fallback은 있지만, `script.hook`이 객체/숫자/null이면 그대로 `escapeHtml(null)` → 문자열 `"null"`이 화면에 찍힌다. `script.cta`가 비면 빈 카드가 뜬다.
→ V1에서는 [Tool Use](https://docs.anthropic.com/en/docs/build-with-claude/tool-use) 또는 JSON mode로 강제하는 게 정석. 지금은 프롬프트로만 강제하는데, 이건 신뢰할 수 없는 신호다.

**(d) `fetch`에 timeout이 없다 — `script.js:98`**
Anthropic API가 30초 행하면 버튼은 영원히 disabled. 발표 중 한 번 일어나면 새로고침해야 한다.
→ `AbortController` + 30초 타임아웃 필수.

**(e) 동시 클릭 race condition — `script.js:309-363`**
"대본 초안 생성"을 빠르게 두 번 누르면 disabled 처리 전 두 번째 핸들러가 또 들어가서 6번 호출된다. 비용도 비용이지만 화면 상태가 꼬인다.
→ 핸들러 첫 줄에서 `if (generateBtn.disabled) return` 체크.

**(f) `apiKeyInput.addEventListener("input", …)`로 키 입력마다 sessionStorage write — `script.js:56-58`**
한 글자 칠 때마다 `sk`, `sk-`, `sk-a` … 가 차례로 저장된다. 큰 문제는 아니지만 `blur` 이벤트로 한 번에 저장하는 게 더 깔끔하다.

**(g) 네이밍 사소한 충돌 — `script.js`**
`TONES[2].id === "hook"`, 그리고 결과 카드 안 라벨도 `Hook · 첫 3초`. 같은 단어가 두 가지 의미(톤 이름 / 카드 섹션)로 쓰여서 디버깅 시 혼란. 톤 id를 `hooky` 또는 `punchy` 같은 이름으로 바꾸면 명확.

**(h) `index.html:20` 주석 vs 실제 클래스 불일치**
```html
<!-- 가로폭을 4xl로 제한해서 가독성 확보 -->
<main class="max-w-3xl mx-auto …">
```
주석은 `4xl`인데 실제는 `max-w-3xl`. 사소하지만 주석은 쉽게 거짓말이 된다.

## 2. UX 결함 5개 (사용자가 처음 화면 봤을 때 헷갈리는 것)

### ① "왜 이 영상인가?" 가 답이 없다
화면 맨 위에 "초간단 5분 양치 루틴, 이거 모르면 손해"가 뜬다. 사용자(=심사위원)의 첫 질문: **"이 영상이 왜 여기에 있죠?"** 답이 없다.

`+39% over avg`만 작은 글씨로 있을 뿐, "이 영상이 왜 트렌딩이고, 왜 우리가 이걸 골랐는지"의 한 줄 설명이 없다. → 데이터 카드 위에 한 줄 캡션 필요: *"카테고리 평균 대비 인게이지먼트 +39% — 오늘의 Top 1"*.

### ② 사용자가 키를 입력하기 전에 결과를 미리 볼 수 없다
첫 화면에서 사용자에게 요구하는 첫 행동이 **"본인 API 키 붙여넣기"**다. 이건 UX 관점에서 큰 마찰이다. 본인 돈을 쓰는 행동을 결과를 본 적도 없는 상태에서 하라는 것.
→ "**예시 결과 보기**" 버튼으로 하드코딩된 샘플 출력을 먼저 보여주고, 그 다음 "내 키로 직접 생성" 흐름이 자연스럽다. 발표 데모로도 이게 더 안정적이다(키 입력 실수, 네트워크 의존을 하나 줄임).

### ③ 3개 탭의 의미를 모른다
"정보형 / 스토리형 / 후킹형" 탭이 결과 영역에 갑자기 등장한다. 사용자가 결과를 받기 전엔 3개 탭이 보이지도 않으니, **결과 받고 나서야 "어, 이게 3개네?"** 가 된다. 이때 사용자는 "셋 중 어떤 걸 써야 하지?"를 모른 채 일단 정보형 탭(기본 활성)을 본다.
→ API 키 입력 영역 위에 "어떤 톤이든 3가지 버전을 동시에 만들어 비교해드립니다" 한 줄 안내. 그리고 결과 받은 후 **"각 톤의 추천 상황"** 한 줄(예: 정보형=B2B/교육, 스토리형=라이프스타일, 후킹형=조회수 우선)이 있으면 의사결정이 가능하다.

### ④ 로딩 상태가 5~15초 동안 "텍스트 한 줄"이다
3톤 병렬 호출은 빠르지만 그래도 보통 5~15초 걸린다. 그동안 사용자는 회색 작은 글씨 "정보형 대본 생성 중…" 만 본다. **발표장에서 이 5~15초가 가장 긴 침묵**이 된다.
→ 스켈레톤 UI(Hook/본문/CTA 카드 모양으로 회색 박스), 또는 단계별 진행 표시("연결 중 → 생성 중 → 파싱 중") 또는 적어도 progress dot 애니메이션. 그리고 톤 하나라도 먼저 도착하면 **그 즉시 보여주기** (현재는 모든 톤이 끝난 뒤에야 화면이 채워짐. `Promise.allSettled` 결과를 한 번에 처리하기 때문 — 각 호출이 끝날 때마다 즉시 렌더하도록 바꾸면 사용자 체감 속도가 3배 빨라진다).

### ⑤ "마음에 안 들면 어떡하지?"의 답이 "전체를 다시 돌리세요"다
사용자가 정보형은 마음에 들고 후킹형만 다시 받고 싶으면, 지금은 "대본 초안 생성" 버튼을 다시 눌러야 한다. **3톤 모두 재생성 = 비용 3배, 시간 3배, 마음에 들었던 정보형도 사라짐**.
→ 각 톤 카드 하단에 "이 톤만 다시 생성" 버튼. 이건 V1 1순위 후보(아래 #5 참고).

## 3. 보안

### sessionStorage 키 관리

`README.md:42`와 `index.html:89`의 문구:
> "키는 브라우저 메모리(sessionStorage)에만 저장되며 **어디로도 전송되지 않습니다.**"

**이 문장은 기술적으로 틀렸다.** 키는 매 호출마다 `api.anthropic.com`으로 전송된다. 사용자가 의미하는 건 "**내 서버로는** 전송되지 않는다"인데, 실제 표현은 "어디로도"다. 발표 때 보안 의식 있는 심사위원이 코드를 열면 즉시 신뢰가 깨진다.
→ 정확한 표현으로 교체: *"키는 사용자 브라우저에만 저장되며, 본 데모 서버는 키를 받거나 저장하지 않습니다. 호출은 사용자 브라우저에서 Anthropic으로 직접 전송됩니다."*

### 진짜 보안 위험: Tailwind CDN의 supply chain 노출

이게 더 중요하다. `index.html:13`:
```html
<script src="https://cdn.tailwindcss.com"></script>
```
이 스크립트는 **사용자 페이지에서 풀 권한으로 실행되며, sessionStorage를 읽을 수 있다.** `cdn.tailwindcss.com`이 한 번이라도 침해되면 모든 방문자의 API 키가 외부로 유출된다. 또한 SRI(`integrity` 속성) 없이 로드되므로 변조 감지조차 불가능.

이건 학생 데모에서 실제로 일어날 가능성은 낮지만, **발표에서 "왜 sessionStorage가 안전합니까?"라는 질문이 들어왔을 때 답할 수 없는 약점**이다.

→ V1에서:
- Tailwind를 빌드해서 정적 CSS로 변경 (CDN 제거).
- 또는 SRI 추가 (`integrity="sha384-..."` + `crossorigin="anonymous"`). 단, Tailwind CDN은 동적 빌드라 SRI가 안 맞을 수 있다.
- 또는 키를 sessionStorage가 아니라 **메모리 변수에만** 보관 (페이지 새로고침 시 다시 입력하게 만든다 — 마찰은 늘지만 supply chain 위험은 0).

### 클라이언트 직접 호출 방식 — 근본적 한계

`anthropic-dangerous-direct-browser-access: true` 헤더 이름 자체가 Anthropic이 "이건 위험하다"고 명시한 것이다. 이유:

1. **요금 폭탄 위험** — 데모 페이지가 검색에 노출되면 누구나 자신의 키로 사용할 수 있고, 만약 발표자가 자기 키를 입력해두고 자리를 떴다가 키가 노출되면 외부 호출을 막을 방법이 없음.
2. **rate limit 절도** — 다른 발표 보러 갔다가 누군가가 본인 키로 생성을 100번 돌리면 본인 quota가 소진된다.
3. **프롬프트가 완전 공개** — 시스템 프롬프트가 그대로 view-source에 노출. 경쟁 가치가 있는 프롬프트는 이렇게 두면 안 됨.

**V0에선 학습용이라 OK이고 README에도 한계로 적혀있긴 하나, 발표 슬라이드에 "V1에선 백엔드 프록시로 키 보호"를 명시적으로 적어두는 게 좋다.** 그래야 심사위원이 "이거 보안상 어떻게 합니까?"라고 물었을 때 "알고 있고, V1 로드맵에 있다"고 답할 수 있다.

### 기타 작은 것
- 동시 클릭 race(섹션 1-e)는 비용 측면에서도 보안 카테고리. 한 번 누르면 키당 3호출인데 두 번 누르면 6호출.
- `apiKeyInput`이 `type="password"`인 건 좋다. 다만 발표 중 화면 공유 상태에서 누가 옆에서 "키가 어떻게 들어가는지 보여줘봐"라고 하면 본인이 무심코 unmask할 위험. 발표 전에 키는 sessionStorage에 미리 넣어두고 입력 자체를 안 하는 게 안전.

## 4. 매니페스토와의 일치도 — "감이 아닌 데이터로 결정"

(매니페스토 원문을 직접 보지 못했지만 사용자가 핵심 가치 #2로 명시한 "감이 아닌 데이터로 결정"을 기준으로 평가)

### 표면적으로는 정합 ✓
- 카테고리 평균 5.91%, 인게이지먼트 8.2%, +39% 같은 숫자가 화면에 노출됨.
- 이 숫자들이 시스템 프롬프트에도 주입됨 (`script.js:71-72`).

### 본질적으로는 불일치 ✗

**V0의 흐름은 다음과 같다:**
```
[입력 데이터: 숫자 + 텍스트]  →  [LLM]  →  [출력: 3개 대본]  →  [사용자가 "감으로" 선택]
                                                                  ↑
                                                             여기서 데이터가 사라짐
```

**문제점 3가지:**

1. **숫자가 의사결정에 영향을 주지 않는다.** 인게이지먼트가 +39%든 +5%든 -10%든, 같은 시스템 프롬프트가 같은 3톤을 만든다. "+39% 카테고리에선 후킹형이 더 잘 먹힌다" 같은 데이터-구동 분기가 없다. 숫자는 **장식**이지 **신호**가 아니다.

2. **출력에 대한 측정 루프가 없다.** 생성된 hook이 8.2% 인게이지먼트 영상에 어울리는지, 더 강한 hook이 필요한지 평가할 방법이 없다. 사용자는 결국 **세 개를 다 보고 마음에 드는 걸 고른다.** 이게 정확히 매니페스토가 막고자 하는 "감으로 결정"이다.

3. **A/B 비교 메커니즘이 없다.** 3개 톤을 동시에 보여주는 것은 비교의 의도지만, **비교의 척도가 없다.** "이 셋 중 어느 게 8.2% 인게이지먼트 영상의 hook으로 더 적합한가?"에 대한 점수/근거가 없다.

### 발표장에서 이 갭이 노출될 시나리오

심사위원: *"이 데모가 trendingshorts의 어떤 핵심 가치를 보여주나요?"*
발표자: *"트렌딩 데이터를 기반으로 대본을 생성합니다."*
심사위원: *"숫자가 +39%라서 결과물이 어떻게 달라졌죠?"*
발표자: *"…"* (달라진 게 없음)

이 순간이 가장 위험하다. **"데이터를 보여주는 것"과 "데이터로 결정하는 것"의 차이를 발표 슬라이드에서 명확히 정리하든가, V1에서 실제 데이터-구동 분기를 만들든가 둘 중 하나는 해야 한다.**

### 즉시 가능한 V0.5 패치 (코드 한 줄 안 바꾸고)
시스템 프롬프트의 `[제약]` 영역에 한 줄 추가:
> "이 영상의 인게이지먼트는 카테고리 평균 대비 +39%로 매우 높습니다. **hook은 이 평균을 넘는 호기심 강도를 가져야 하며, 본문 첫 번째 단계는 첫 5초 내 시청자 이탈을 막을 강한 보상을 약속해야 합니다.**"

이러면 +39% 숫자가 실제로 출력 형태에 영향을 주는 신호로 바뀐다. 코드 변경은 불필요하고 프롬프트만 수정.

## 5. V1 우선순위

### 추가해야 할 것 #1: **각 톤별 "데이터 적합도 점수"**

매니페스토와의 갭을 메우는 가장 강력한 한 수. 동작 방식:

- 3개 대본 생성 후, **별도의 추가 호출**로 Claude에게 평가 요청:
  > "다음 3개 hook 중 인게이지먼트 +39% 카테고리 영상에 가장 적합한 것은? 각 hook을 호기심 강도/카테고리 적합성/이탈 방지력 3축에서 0-10 채점하고 1순위를 추천하세요."
- 결과를 각 카드 상단에 점수 뱃지로 표시. **추천 1순위 카드는 시각적으로 강조** (테두리 색, "추천" 뱃지).

이게 만드는 변화:
- 사용자(/심사위원)가 "정보형/스토리형/후킹형 중 뭘 골라요?"의 답을 데이터로 받음.
- 매니페스토 #2 "감이 아닌 데이터" 정합 ✓
- 발표 데모 "스토리텔링"이 강해짐 ("→ 8.2% 인게이지먼트 영상엔 후킹형 8.5점이 가장 적합. 이게 우리가 추천하는 안입니다.")
- 추가 비용은 톤당 약 200토큰 짜리 평가 호출 1회 → 매우 저렴

이건 단순 기능 추가가 아니라 **제품의 가치 명제(value proposition)를 V0의 "3개 만들어줌" → V1의 "데이터 기반으로 1개를 추천함"으로 격상시키는 변화**다.

### 빼도 되는 것 #1: **3톤 병렬 생성**

이걸 빼라는 게 아니라 **V0의 디폴트 동작에서는 빼라**는 것. 이유:

1. **발표 현장의 단일 최대 위험.** 3개 병렬 호출 = 실패 surface 3배. 하나만 401/429/timeout 떠도 화면에 빨간 카드. 발표용 Wi-Fi에서 가장 위험.
2. **비용 3배** — 학생 키로 데모를 여러 번 반복하면 부담.
3. **사용자의 인지 부하.** 결과를 받자마자 3개 탭을 한꺼번에 봐야 한다. 사용자(심사위원)의 첫 30초가 분산됨.

**대신:**
- 디폴트는 "정보형 1개" 생성.
- "다른 톤도 보기" 토글 또는 "이 톤도 추가 생성" 버튼.
- 이렇게 바꾸면 발표 흐름이 *"입력 → 1개 결과 → '다른 톤 어때?' 클릭 → 비교"* 의 **스토리**가 된다. 지금은 *"버튼 → 침묵 5초 → 갑자기 3개"* 의 데이터 덤프.

이 변경이 V1 추가(#1 데이터 적합도 점수)와 합쳐지면 흐름은:
1. 입력 → 정보형 1개 생성 (5초)
2. "다른 톤도 보기" 클릭 → 스토리형/후킹형 추가 생성 (5초)
3. **자동으로 적합도 점수 호출 → 추천 톤 강조 (3초)**

**이게 매니페스토 정합 + 발표 안정성 + 사용자 인지 부하 셋 다 잡는 변경이다.**

## 결론

V0는 **"잘 만든 학생 과제"의 기준은 충분히 통과**한다. 코드는 깔끔하고, 의도는 명확하고, README는 충실하다.

하지만 **"발표장에서 매니페스토를 증명하는 데모"의 기준은 아직**이다. 표면(데이터 노출)과 본질(데이터로 결정)이 어긋나 있고, 외부 의존성 3개가 동시에 살아있어야 화면이 도는 구조라 발표 현장 변수에 약하다.

**6주 동안 1개만 한다면: V1 추가 #1 (데이터 적합도 점수 호출)**.
이게 매니페스토 갭과 발표 스토리 양쪽을 동시에 풀어준다.

**6주 동안 1개만 뺀다면: 디폴트 3톤 병렬 → 1톤 + 추가 버튼**.
이게 발표 안정성을 1순위로 끌어올린다.

남은 5주는 사용성 개선(UX 결함 5개)과 코드 견고화(섹션 1의 a-e)에 분배.
