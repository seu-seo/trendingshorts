# PREMORTEM.md
### Shortform Pulse — 프리모텀 분석

---

## 이 문서를 쓰는 이유

프리모텀은 시작하기 전에 "이미 실패했다고 가정하고 왜 실패했는지"를 역으로 분석하는 방법이다. 낙관 편향을 제거하고 실제로 마주칠 수 있는 문제들을 미리 꺼내두는 것이다.

이 버전(v3)은 피치덱 범위가 좁혀지면서 새로 생각해봐야 할 리스크가 달라졌다. 로드맵과 수익화 관련 리스크보다는 핵심 기능 4개가 실제로 유효한지, API 연동이 가능한지, 크리에이터가 실제로 쓰는지에 집중했다.

---

## 가정

> **Shortform Pulse를 출시하고 3개월이 지났다.**
> **초기 사용자 몇 명이 있었지만 대부분 이탈했다.**
> **에이카랩스 파일럿은 제대로 시작되지 못했다.**
> **툴이 있는데 아무도 쓰지 않는 상황이 됐다.**
> **정확히 어느 지점에서 잘못됐는지 팀이 모른다.**

## 실패 시나리오 분석

---

### �� 시나리오 1 · TikTok API를 실제로 쓸 수 없었다

**가상 사후 평가**
```
"TikTok Display API 신청을 했는데 승인이 수 개월째 안 났다.
한국 개발자 계정으로 접근이 제한되는 조건이 있었다.
결국 3개 플랫폼 중 하나가 빠진 채로 운영해야 했다.
'3개 플랫폼 통합'이라는 핵심 가치가 무너졌는데,
그걸 사전에 확인하지 않았다."
```

**왜 이렇게 됐는가**

TikTok Display API는 YouTube Data API나 Instagram Graph API와 달리 접근 조건이 까다롭다. 심사 기간이 불확실하고 한국 시장에서 개발사 계정을 만드는 과정도 복잡하다. 이 리스크를 체크하지 않고 "3개 플랫폼 통합"을 전제로 제품을 설계하면, 실제 구현에서 막혔을 때 대응이 어렵다.

TikTok MAU가 1.9B(DemandSage, 2026)이고 평균 인게이지먼트가 3.70%(Social Insider, 2025)라는 데이터를 쓰고 있는데, 이 플랫폼 데이터를 실제로 수집하지 못하면 피치덱의 핵심 전제가 흔들린다.

**예방책**
- 개발 시작 전에 TikTok Display API 한국 계정 접근 조건과 심사 절차를 먼저 확인한다
- 승인이 지연되는 경우를 위한 fallback 방안을 준비해둔다 (스크래핑 등)
- MVP는 YouTube 하나로 먼저 가치 증명을 하고 TikTok·Instagram은 순차적으로 붙인다

### �� 시나리오 2 · 인사이트 데이터가 크리에이터에게 실제로 유용하지 않았다

**가상 사후 평가**
```
"Creator Insights 기능을 만들어서 최적 업로드 시간,
영상 길이, 포맷 데이터를 보여줬는데, 실제 크리에이터들이
'이미 다 알고 있는 내용이에요'라고 했다.
우리가 새롭다고 생각한 게 이미 YouTube 크리에이터 스튜디오나
무료 블로그에 다 있는 내용이었다."
```

**왜 이렇게 됐는가**

크리에이터가 실제로 필요한 정보와 우리가 제공하는 정보가 다를 수 있다. 최적 업로드 시간이나 영상 길이는 이미 많은 곳에서 다루는 주제다. "크로스플랫폼 비교"가 실제로 우리가 제공할 수 있는 차별점인데, 그것만으로 충분한지 검증하지 않았다.

**예방책**
- 출시 전에 크리에이터 5명 이상을 인터뷰해서 실제로 필요한 데이터가 무엇인지 확인한다
- YouTube 크리에이터 스튜디오, TikTok Analytics에서 이미 제공하는 것과 우리가 추가하는 것을 명확히 구분한다
- Cross-Platform Compare처럼 기존 툴에서 할 수 없는 기능을 핵심으로 밀고, 나머지는 보조로 배치한다

---

### �� 시나리오 3 · AI 소재 생성 결과를 아무도 쓰지 않았다

**가상 사후 평가**
```
"에이카랩스 담당자가 AI 소재를 보고 '이건 우리 톤이 아닌데요'라고 했다.
직접 고쳐서 올렸고, 다음부터는 버튼을 안 눌렀다.
결국 트렌드 탐색만 하다가 그것도 안 하게 됐다."
```

**왜 이렇게 됐는가**

AI 소재 생성 품질은 브랜드 컨텍스트의 깊이에 달려 있다. 제품명, 타겟, 톤 정도만 입력받으면 어느 브랜드에나 붙을 수 있는 범용 카피가 나온다. 에이카랩스 담당자가 그걸 보고 쓸 이유를 못 느끼면 그냥 안 쓴다.

**예방책**
- 브랜드 설정에 RTB(저불소 ppm, 미함유 성분, 치과의사 추천 여부), 절대 쓰지 않는 표현, 잘 됐던 과거 소재 예시를 추가한다
- 생성된 소재에 마케터가 피드백을 남기면 재생성되는 루프를 만든다
- 첫 3개 소재는 팀이 직접 수정해서 완성도를 높이고, 그 결과를 프롬프트 개선에 반영한다

---

### �� 시나리오 4 · 에이카랩스 파일럿에서 성과 데이터가 쌓이지 않았다

**가상 사후 평가**
```
"소재를 만들었는데 에이카랩스가 집행을 안 했다.
내부 사정이 있었다. A→A'→B→KPI 파이프라인에
데이터가 하나도 없었다. '마케터에게도 유효하다'는
가설을 검증할 방법이 없어졌다."
```

**왜 이렇게 됐는가**

외부 팀이 만든 툴을 내부 마케터의 업무 흐름에 끼워 넣는 것은 항상 저항이 있다. 소재를 만들어놔도 집행까지 가려면 내부 결재, 예산, 타이밍이 맞아야 한다.

**예방책**
- 파일럿 시작 전에 "이 기간에 이 소재를 집행한다"는 커밋을 에이카랩스로부터 받아둔다
- 팀원 한 명이 에이카랩스와 주 1회 싱크를 갖고 진행 상황을 확인한다
- 초반 2주는 팀이 직접 소재를 만들어 제공해서 담당자 부담을 최소화한다

---

### �� 시나리오 5 · 경쟁 툴이 이미 충분했다

**가상 사후 평가**
```
"출시 후 찾아보니 Exploding Topics, Vidooly, TrendTok 같은
툴들이 이미 있었다. '크로스플랫폼 비교'는 우리가 차별점이라고
생각했는데, 다른 방식으로 이미 하고 있는 툴들이 있었다.
사용자를 바꾸게 만드는 게 생각보다 훨씬 어려웠다."
```

**왜 이렇게 됐는가**

경쟁 툴 분석을 충분히 하지 않으면 우리가 새롭다고 생각하는 기능이 이미 존재할 수 있다. 특히 "한국 크리에이터·마케터 특화"가 실제로 차별점이 되려면 언어, 카테고리, UX가 실제로 다르게 느껴져야 한다.

**예방책**
- 출시 전에 주요 경쟁 툴 5개를 직접 써보고 기능 비교표를 만든다
- 기존 툴이 제공하지 않는 기능 2~3가지를 명확히 정의하고, 그것만 먼저 만든다
- "한국어 기반, 한국 크리에이터 특화 카테고리"를 초기부터 명확한 포지셔닝으로 가져간다

---

### �� 시나리오 6 · 실시간 수집 비용을 감당하지 못했다

**가상 사후 평가**
```
"API 호출 비용과 서버 비용이 사용자가 늘수록 같이 올라갔다.
초기에 비용 계획을 제대로 세우지 않아서,
세 달이 지나자 운영 비용이 감당이 안 되기 시작했다."
```

**왜 이렇게 됐는가**

실시간 트렌드 수집은 API 호출 횟수와 데이터 저장 비용이 함께 올라간다. YouTube Data API는 일일 할당량(quota)이 있고, 많은 카테고리를 자주 수집하면 유료 구간으로 진입한다. 초기에 이 계산을 하지 않으면 나중에 서비스를 줄이거나 중단해야 하는 상황이 온다.

**예방책**
- 개발 시작 전에 사용자 규모별 월간 API 비용을 시뮬레이션한다 (100명/1,000명/10,000명)
- 무료로 제공하는 데이터 수집 범위의 한도를 명확히 정한다
- 비용이 수익보다 앞서는 구간의 기간과 한도를 팀이 합의해둔다

---

### �� 시나리오 7 · 크리에이터가 실제로 필요한 게 우리가 만든 것과 달랐다

**가상 사후 평가**
```
"출시 후 인터뷰해봤더니 크리에이터들은
트렌드 데이터보다 '어떤 썸네일이 클릭률이 높나'를
더 궁금해했다. 우리가 가정으로 설계한 기능들이
실제 페인포인트와 거리가 있었다."
```

**왜 이렇게 됐는가**

팀 배경이 마케팅·데이터·기획·비즈니스라 크리에이터 경험이 있는 팀원이 없다. 크리에이터가 실제로 무엇을 불편해하는지 직접 물어보지 않으면 우리의 가정이 맞는지 알 수 없다. 크리에이터 90%가 번아웃을 겪는다는 데이터가 있는데, 그 번아웃의 구체적인 원인이 우리가 해결하려는 것과 일치하는지 확인해야 한다.

**예방책**
- 개발 전에 크리에이터 5명 이상을 인터뷰한다. 실제 불편함, 지금 쓰는 툴, 돈을 낼 기준을 확인한다
- 초기 베타 사용자 10명을 모집해서 주 1회 피드백을 받는다
- 기능을 추가할 때마다 "크리에이터가 실제로 이걸 원하는가"를 확인하는 루프를 만든다

## 리스크 우선순위 요약

| 시나리오 | 발생 가능성 | 영향도 | 우선순위 |
|---------|-----------|--------|---------|
| TikTok API 접근 제한 | 높음 | 높음 | 🔴 개발 전 확인 필수 |
| Creator Insights 유용성 부족 | 중간 | 높음 | 🔴 인터뷰로 사전 검증 |
| AI 소재 품질 → 미사용 | 높음 | 중간 | 🔴 브랜드 컨텍스트 강화 |
| 에이카랩스 파일럿 데이터 없음 | 중간 | 중간 | 🔴 사전 커밋 필요 |
| 경쟁 툴과 차별화 부재 | 중간 | 높음 | 🟡 경쟁 분석 먼저 |
| 인프라 비용 초과 | 중간 | 중간 | 🟡 비용 시뮬레이션 |
| 크리에이터 니즈 미파악 | 중간 | 높음 | 🟡 인터뷰 먼저 |

상위 4개는 개발 시작 전에 반드시 확인하거나 결정해야 한다. 특히 TikTok API는 기술적 실현 가능성의 문제라 가장 먼저 봐야 한다.

## 팀 합의 액션

```
[ ] 1. API 사전 조사 (개발 전 필수)
        TikTok Display API 한국 계정 접근 조건 확인
        YouTube Data API 일일 할당량 및 비용 계산
        Instagram Graph API Reels 데이터 접근 범위 확인
        → 담당: 데이터팀 / 기한: 개발 착수 전

[ ] 2. 크리에이터 인터뷰 최소 5건
        실제 불편함, 현재 사용 툴, 지불 의사 확인
        번아웃 원인과 우리 해결책이 맞닿아 있는지 검증
        → 담당: 마케팅팀 + 기획팀 / 기한: MVP 설계 전

[ ] 3. 경쟁 툴 분석
        주요 경쟁 툴 5개 직접 사용 후 기능 비교 문서화
        Shortform Pulse만의 차별점 2-3개 확정
        → 담당: 마케팅팀 / 기한: 랜딩 페이지 제작 전

[ ] 4. 인프라 비용 시뮬레이션
        사용자 규모 시나리오별 월간 비용 계산
        무료 제공 범위의 한도 설정
        → 담당: 데이터팀 / 기한: 개발 착수 전

[ ] 5. 에이카랩스 파일럿 커밋
        집행 기간, 소재 수, 담당자 확정
        주간 싱크 일정 확정
        → 담당: PM / 기한: 소재화 레이어 개발 전

[ ] 6. 브랜드 컨텍스트 강화
        에이카랩스 RTB, 금지 표현, 레퍼런스 소재 수집
        프롬프트에 반영
        → 담당: 마케팅팀 / 기한: 소재화 레이어 개발 전
```

## 마지막으로

v3로 오면서 로드맵과 수익화 관련 내용이 피치에서 빠졌다. 제품의 범위를 좁히고 핵심 기능 4개에 집중하는 방향이다. 그에 맞게 리스크도 재정렬했다. 지금 가장 중요한 건 TikTok API 접근 가능 여부와 크리에이터가 실제로 이 툴을 필요로 하는지다.

90%+가 구독자 1만을 못 넘고, 90%가 번아웃을 겪는다는 데이터가 있다. 이 문제를 해결하는 툴이라는 전제가 맞는지, 우리가 만드는 것이 실제로 그 원인을 건드리는지 — 인터뷰와 파일럿이 이걸 확인해주는 과정이다.
