# PREMORTEM.md
### Shortform Pulse — 프리모텀 분석

---

## 이 문서를 쓰는 이유

프리모텀은 시작하기 전에 "이미 실패했다고 가정하고 왜 실패했는지"를 역으로 분석하는 방법이다. 낙관 편향을 제거하고 실제로 마주칠 수 있는 문제들을 미리 꺼내두는 것이다.

이 버전은 SPEC.md를 반영해서 리스크를 재정렬했다. 제품의 세 가지 핵심 차별점 — ① 정보 비대칭 해소, ② 서브카테고리 단위 추천, ③ 대본 초안까지의 워크플로우 완성 — 이 각각 실제로 작동하는지를 중심으로 리스크를 본다.

> **v8 업데이트 (2026-06-15):** 아래 팀 합의 액션 중 일부가 feat/v8-next.js에서 구현 완료됨. 해결된 항목에 ✅ 표기.

---

## 가정

> **Shortform Pulse를 출시하고 3개월이 지났다.**
> **초기 사용자 몇 명이 있었지만 대부분 이탈했다.**
> **에이카랩스 파일럿은 제대로 시작되지 못했다.**
> **툴이 있는데 아무도 쓰지 않는 상황이 됐다.**
> **정확히 어느 지점에서 잘못됐는지 팀이 모른다.**

### 🔴 시나리오 1 · TikTok / Instagram 데이터를 실제로 쓸 수 없었다

**가상 사후 평가**
```
"Apify 유료 플랜을 활성화했는데 Instagram Reels와 TikTok
메타데이터 수집이 생각보다 불안정했다.
한국 계정 기준 데이터 신선도가 떨어졌고,
YouTube만 제대로 도는 상태에서 데모를 했다.
'3개 플랫폼 크로스 비교'라는 핵심 가치가 사라졌다."
```

**왜 이렇게 됐는가**

SPEC.md에서 데이터 수집 계획이 명시돼 있다. YouTube Data API v3는 공식 무료 한도 내에서 일 1회 갱신, Instagram과 TikTok은 Apify 유료 플랜으로 주 1회 갱신이다. Apify 스크래퍼는 플랫폼 구조 변경에 민감하고 수집 안정성이 공식 API 대비 낮다. 이 의존성을 사전에 검증하지 않으면 데모에서 막힌다.

**예방책**
- Apify Instagram·TikTok Scraper를 개발 착수 전에 한국 계정 기준으로 실제 수집 테스트한다
- 외부 API 장애 시 Mock fallback으로 데모가 유지되도록 설계한다 (SPEC 11번 항목)
- MVP는 YouTube 단일 플랫폼으로 가치 증명 후 Instagram·TikTok을 순차적으로 붙인다

### 🔴 시나리오 2 · 서브카테고리 분류가 실제로 유용하지 않았다

**가상 사후 평가**
```
"Claude 기반 서브카테고리 자동 분류기를 만들었는데,
분류 결과가 직관적이지 않았다.
'먹방 → 편의점 리뷰'가 아니라 '음식 콘텐츠 → 유통 채널 리뷰'
같은 식으로 나왔다. 크리에이터들이 보고
'이게 나한테 맞는 카테고리인지 모르겠다'고 했다."
```

**왜 이렇게 됐는가**

서브카테고리 단위 추천은 Shortform Pulse의 핵심 차별점이다. 그런데 SPEC 9.2에서 "카테고리 ID 통합 표준 합의"가 아직 미완이라고 명시되어 있다. 팀원별로 카테고리 체계가 다르다(지은 6개, 승연 8개, 규동 12+15개). 이 상태에서 분류기를 만들면 사용자에게 일관성 없는 서브카테고리가 노출된다.

**예방책**
- 카테고리 ID 통합 표준을 개발 착수 전에 팀 전체가 합의한다 (SPEC 9.2 최우선 항목)
- 서브카테고리 명칭은 크리에이터가 실제로 쓰는 언어로 정의한다 (인터뷰 선행)
- Claude 분류기 결과를 크리에이터 5명에게 보여주고 직관성을 검증한다

### 🔴 시나리오 3 · 대본 초안이 아무도 쓰지 않는 결과물이었다

**가상 사후 평가**
```
"대본 초안을 생성해줬는데 크리에이터들이
'이건 내 채널 톤이 아니에요'라고 했다.
HOOK이 너무 일반적이고, 내 팔로워가 반응할 언어가 아니었다.
한 번 써보고 다시는 제작 탭을 열지 않았다."
```

**왜 이렇게 됐는가**

SPEC 6.3에서 대본 생성 컨텍스트(페르소나·카테고리·트렌드 메타데이터 중 어떤 것을 어떻게 활용할지)가 TBD로 남아 있다. 컨텍스트 없이 트렌드 키워드만 넣으면 어느 채널에도 붙을 수 있는 범용 대본이 나온다. 범용 대본은 크리에이터가 쓰지 않는다.

대본 초안은 Shortform Pulse가 "무엇이 뜨는가"에서 멈추지 않고 "오늘 뭘 찍어야 하는가"까지 완성한다는 핵심 가치의 실현체다. 이게 작동하지 않으면 기존 툴과의 차별점이 사라진다.

**예방책**
- 대본 생성 컨텍스트 입력 합의를 제작 탭 개발 전에 팀이 완료한다 (SPEC 9.2)
- 페르소나 설정 정보(카테고리, 콘텐츠 스타일)를 대본 생성 프롬프트에 반드시 반영한다
- 크리에이터의 기존 채널 톤·포맷 정보를 입력할 수 있는 필드를 추가한다
- 첫 3개 대본은 팀이 직접 수정해서 완성도를 검증하고 프롬프트에 반영한다

### 🔴 시나리오 4 · 에이카랩스 파일럿에서 성과 데이터가 쌓이지 않았다

**가상 사후 평가**
```
"소재를 만들었는데 에이카랩스가 집행을 안 했다.
내부 사정이 있었다. A→A'→B→KPI 파이프라인에
데이터가 하나도 없었다. '마케터에게도 유효하다'는
가설을 검증할 방법이 없어졌다."
```

**왜 이렇게 됐는가**

외부 팀이 만든 툴을 내부 마케터의 업무 흐름에 끼워 넣는 것은 항상 저항이 있다. 소재를 만들어놔도 집행까지 가려면 내부 결재, 예산, 타이밍이 맞아야 한다.

**예방책**
- 파일럿 시작 전에 "이 기간에 이 소재를 집행한다"는 커밋을 에이카랩스로부터 받아둔다
- 팀원 한 명이 에이카랩스와 주 1회 싱크를 갖고 진행 상황을 확인한다
- 초반 2주는 팀이 직접 소재를 만들어 제공해서 담당자 부담을 최소화한다

### 🔴 시나리오 5 · API 계약 미합의 상태에서 통합이 깨졌다

**가상 사후 평가**
```
"각자 만든 모듈을 합치려는데 데이터 형식이 달랐다.
카테고리 ID가 팀원마다 달랐고,
페르소나 정보가 대본 생성 탭까지 전달되지 않았다.
Onboarding → Dashboard → Production 흐름이
실제로는 연결되지 않은 채 데모를 했다."
```

**왜 이렇게 됐는가**

SPEC 9.2에서 명시된 TBD 항목들이 합의 없이 통합 단계에 진입하면 이 시나리오가 발생한다. 특히 페르소나 → 추천 → 대본 생성으로의 컨텍스트 전달 방식이 미합의 상태라면, 세 탭이 각각 동작하더라도 하나의 흐름이 되지 않는다.

**예방책**
- SPEC 9.2의 TBD 항목을 통합 단계 착수 전에 팀 전체가 완료한다:
  - 페르소나 분석 엔드포인트 명세
  - 대본 생성 엔드포인트 명세
  - TrendItem 통합 스키마
  - 카테고리 ID 통합 표준
  - 페르소나 → 추천 → 대본 생성 컨텍스트 전달 방식
- `GET /api/trends` 응답 형식을 먼저 합의하고, 그 위에 각 모듈을 붙인다

### 🟡 시나리오 6 · 정보 비대칭 해소라는 프레임이 크리에이터에게 와닿지 않았다

**가상 사후 평가**
```
"'내 피드는 내 버블만 보여준다'는 설명을 했는데
크리에이터들이 잘 이해하지 못했다.
그냥 '트렌드 알려주는 툴이네요'로 받아들였다.
우리가 생각한 차별점이 사용자에게 전달되지 않았다."
```

**왜 이렇게 됐는가**

정보 비대칭 프레임은 제품의 존재 이유를 날카롭게 설명하지만, 크리에이터가 직접 경험하는 언어와 다를 수 있다. "알고리즘의 역설"보다 "오늘 뭘 찍을지 정하는 데 몇 분 걸려요?"가 더 빠르게 공감을 만들 수 있다.

**예방책**
- 크리에이터 인터뷰에서 "오늘 찍을 것을 정하는 데 얼마나 걸리나", "어디서 아이디어를 얻나"를 직접 묻는다
- 크리에이터가 실제로 쓰는 언어로 문제를 프레이밍한다 (30~90분 의사결정 소요 시간이 더 직접적일 수 있다)
- 정보 비대칭 설명은 투자자·피치 맥락에서 유지하고, 크리에이터용 랜딩은 다른 언어를 쓴다

### 🟡 시나리오 8 · 실시간 수집 비용을 감당하지 못했다

**가상 사후 평가**
```
"Apify 유료 플랜과 YouTube API 호출 비용, 서버 비용이
사용자가 늘수록 같이 올라갔다. 세 달이 지나자
운영 비용이 감당이 안 되기 시작했다."
```

- Apify 유료 플랜 비용과 수집 빈도(주 1회)가 실제 서비스 품질에 충분한지 검증한다

### 🟡 시나리오 9 · 크리에이터가 실제로 필요한 게 우리가 만든 것과 달랐다

**가상 사후 평가**
```
"출시 후 인터뷰해봤더니 크리에이터들이
'오늘 뭘 찍을지보다 올린 영상이 왜 안 뜨는지가 더 급해요'
라고 했다. 제작 전 의사결정보다 사후 분석이 더 필요했다."
```

- 개발 전에 크리에이터 5명 이상을 인터뷰한다. "오늘 찍을 것을 정하는 과정"을 구체적으로 묻는다
- 현재 30~90분이라는 베이스라인 수치가 실제로 맞는지 검증한다
- 크리에이터가 "오늘 뭘 찍을지"를 가장 큰 문제로 느끼는지 확인한다

## 팀 합의 액션

```
[ ] 1. Apify 수집 테스트 (개발 전 필수)
        Instagram Reels·TikTok 한국 계정 기준 실제 수집 테스트
        데이터 신선도·안정성 확인
        Mock fallback 설계 확정  ← DISABLE_APIFY=true 구현 완료 ✅
        → 담당: 데이터팀 / 기한: 개발 착수 전

[✅] 2. SPEC 9.2 TBD 항목 팀 합의 (v8에서 단일 구현으로 해소)
        [✅] 카테고리 ID 통합 표준 확정
             → food|beauty|lifestyle|edu|gaming|fitness|art (7개)
             OnboardingPrefsScreen·TrendsScreen·API 모두 동일 사용
        [✅] 페르소나 분석 엔드포인트 명세 → POST /api/persona
        [✅] 대본 생성 엔드포인트 명세 → POST /api/conti, POST /api/generate
        [✅] TrendItem 통합 스키마 → lib/types.ts Trend 타입
        [✅] 페르소나 → 추천 → 대본 컨텍스트 전달 방식
             → page.tsx answers/prefs/personaResult prop 드릴다운

[ ] 3. 크리에이터 인터뷰 최소 5건
        "오늘 찍을 것을 정하는 과정" 구체적으로 확인
        30~90분 베이스라인 수치 검증
        서브카테고리 분류 결과 직관성 검증
        → 담당: 마케팅팀 + 기획팀 / 기한: MVP 설계 전

[✅] 4. 대본 생성 컨텍스트 합의 (v8에서 구현 완료)
        [✅] 페르소나·카테고리·트렌드 메타데이터 → POST /api/conti body에 포함
        [✅] 트렌드 선택 → Production 화면 직행 (마운트 즉시 생성)

        Shortform Pulse만의 차별점 2-3개 확정
```

## 마지막으로

Shortform Pulse의 핵심 가치는 세 가지로 압축된다. 정보 비대칭 해소, 서브카테고리 단위 추천, 대본 초안까지의 워크플로우 완성. 이 세 가지가 모두 실제로 작동할 때 "기존 툴은 무엇이 뜨는가까지만 알려준다. 우리는 오늘 뭘 찍어야 하는가까지 완성한다"는 포지셔닝이 성립한다.

지금 가장 중요한 건 API 계약 합의(SPEC 9.2)와 데이터 수집 안정성 검증이다. 세 탭이 각각 동작해도 하나의 흐름으로 연결되지 않으면 차별점이 사라진다.
